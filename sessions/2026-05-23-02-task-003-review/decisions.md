# Решения — task-003-review

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | Раскладка compose-файлов: база + `override.yml` (dev, в репо) + `prod.yml` (явный `-f`) | (б) два полных файла `docker-compose.yml` (dev) и `docker-compose.prod.yml` (prod) под явный `-f` каждый | Стандартная схема compose v2. Минимум дублирования (база описывает все сервисы один раз), понятная инверсия (dev — авто-подхват, prod — явный). Реализация поэтапно: сейчас только база (db+redis), `override.yml` и `prod.yml` появятся в TASK-026 вместе с `bot`/`web` сервисами и Dockerfile-ами | [`docs/07-deployment.md`](../../docs/07-deployment.md), [`state/DECISIONS.md`](../../state/DECISIONS.md) |
| 2 | `.env.example` показывает оба варианта `DATABASE_URL` / `REDIS_URL` (localhost / compose-имена); dev (localhost) — по умолчанию | (б) отложить до TASK-004 с автодетекцией через config-layer; (в) ввести отдельные переменные `DATABASE_URL_DEV` / `DATABASE_URL_DOCKER` | (б) откладывает решение проблемы, в которую разработчик упирается прямо сейчас. (в) усложняет config-layer ради проблемы документации. Простой комментарий с двумя примерами решает задачу в одно касание | [`infra/.env.example`](../../infra/.env.example), [`state/DECISIONS.md`](../../state/DECISIONS.md) |
| 3 | `make nuke` оставлен с подтверждением через ввод `NUKE` | `y/yes` (слабо), `--force` флаг (легко проставить случайно), `DELETE` (путается с SQL) | Слово `NUKE` достаточно сильное и непривычное для повседневного набора. Шесть нажатий — небольшая цена за нежелание удалить данные случайно | [`Makefile`](../../Makefile), [`state/DECISIONS.md`](../../state/DECISIONS.md) |
| 4 | Старый черновик compose в `docs/07-deployment.md` заменён на актуальную форму со ссылкой на реальный файл | Оставить черновик «как ориентир» | Черновик расходился с реальностью и сбивал бы будущих читателей. Лучше иметь ссылку на источник истины + заготовку будущей prod-формы | [`docs/07-deployment.md`](../../docs/07-deployment.md) |
