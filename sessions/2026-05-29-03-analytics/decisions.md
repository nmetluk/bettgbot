# Решения — analytics

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | Набор метрик: (1) динамика прогнозов по дням, (2) точность по категориям, (3) воронка регистрация→первый прогноз, (4) топ-события по числу прогнозов | Больше метрик сразу | Ровно то, что в roadmap-карточке; держим объём M | `TASK-059` |
| 2 | Графики — **Chart.js** с `cdn.jsdelivr.net` для (1) line и (2) bar; (3)/(4) — таблицы/CSS-бары | CSS-only бары; серверный SVG; ECharts | Chart.js: canvas, **без `eval`** → совместим с текущим CSP (script-src уже разрешает jsdelivr, style-src имеет `unsafe-inline`) — менять CSP не нужно. Богаче CSS-баров, легче серверного SVG | `TASK-059` |
| 3 | Окно динамики по дням — последние 30 дней (дефолт), нулевые дни заполняются | С начала проекта; без заполнения дыр | 30 дней читаемо; заполнение нулей даёт ровную линию | `TASK-059` |
| 4 | Точность по категориям — `correct/resolved` на JOIN Prediction→Event→Category, тот же filtered-count паттерн | Хранить агрегат | Read-only агрегат достаточно быстр на текущих объёмах; без денормализации | `TASK-059` |
| 5 | Воронка — `distinct предсказавших / всего незаблокированных` (+ %) | По активным за 30д | Простая и понятная конверсия «дошёл до первого прогноза» | `TASK-059` |
| 6 | Скоуп — только админ-экран `/analytics`, read-only | + экспорт CSV; + бот | Карточка про интерфейс админки; экспорт/бот — отдельные будущие задачи | `TASK-059` |
| 7 | Без новой модели/миграции | Таблица-агрегат/материализованное представление | Всё считается на лету из существующих таблиц | `TASK-059` |

ADR не нужен: переиспользуются модель, паттерн агрегации и дизайн-система; единственная новая
зависимость — Chart.js (CDN, без изменения CSP), что не является архитектурной развилкой уровня ADR.
