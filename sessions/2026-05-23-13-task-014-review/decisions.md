# Решения — task-014-review

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | Back-target в FSM прогноза остаётся «к категории», даже если пользователь вошёл в FSM из «Мои прогнозы» | Расширить три prediction-callback'а полем `back_my_tab: MyTab \| None`; хранить в FSM-data с сериализацией | Расширение трёх CB-классов увеличивает поверхность и риски с лимитом 64 байта callback_data. Хранение в FSM-data требует учёта истечения state. Потеря UX (после отмены «Изменить прогноз» из «Мои прогнозы» возврат не туда) — мелкая, прецедентно нечастая (пользователь сам сначала нажал кнопку «🔙 К моим прогнозам», доберётся). Фиксируем в BACKLOG как тех-долг — реальный сигнал из UX-наблюдений будет триггером | [`state/BACKLOG.md`](../../state/BACKLOG.md) (тех-долг), [`state/DECISIONS.md`](../../state/DECISIONS.md) |
| 2 | Текст кнопки прогноза в списке — только `event.title`, без статус-эмодзи | Добавить `⏳/✅/❌` к началу текста кнопки | Информация о статусе **уже** есть в тексте под кнопкой (`MY_ROW_ARCHIVE` содержит `status_emoji`). Дублирование в кнопке загружает UI, кнопочный текст коротким должен оставаться (лимит Telegram + читаемость) | [`src/bot/keyboards/__init__.py`](../../src/bot/keyboards/__init__.py) |
| 3 | Маркер `✓` активного таба остаётся | Менять только заголовок body, без маркера | Явный визуальный сигнал «вы здесь». Минимальная стоимость в 2 символа на кнопке. Без него пользователь должен догадаться по заголовку, что менее очевидно | [`src/bot/keyboards/__init__.py`](../../src/bot/keyboards/__init__.py) |
| 4 | `allow_archived: bool = False` в `render_event_card` принимаем как штатное решение | Убрать guard `is_archived` целиком (`predictions_close_at` и так не пройдёт для архива → кнопка прогноза не покажется) | Текущий guard защищает от гонки: админ архивировал событие, пока пользователь видел его в каталоге → каталог не должен показывать архивную карточку. `allow_archived=True` — явный whitelist для входа из «Мои прогнозы → Архив», где архивное событие — норма. Дизайн чистый и контракт `render_event_card` явный | [`src/bot/routers/events.py`](../../src/bot/routers/events.py) |
| 5 | Регресс-тесты на сборку `back_button` в `on_event` / `on_predict_cancel` не добавляем | Добавить тесты, которые `assert_called_with(back_button=("...", CategoryCb(...)))` | Существующие тесты прошли без правок — они проверяют поведение через `query.message.edit_text.assert_awaited_once` и сервис-моки, не лезут в детали клавиатуры. Явные тесты на структуру `back_button` — это тесты на код, а не на поведение. Если в будущем поведение сломается (back-кнопка ведёт не туда) — это поймает либо ручное тестирование, либо новый тест на конкретный сценарий | [`tests/unit/bot/routers/test_events_handler.py`](../../tests/unit/bot/routers/test_events_handler.py) |
| 6 | `event_card_kbd` и `render_event_card` принимают `back_button: tuple[str, CallbackData]` вместо `back_category_id: int \| None` — оставляем как есть | Выделить `BackButton` как `NamedTuple` / `dataclass` | Один параметр, не передаётся через многие слои. Tuple короче и без дополнительного типа. Если в TASK-015+ появится четвёртая точка вызова — рассмотрим выделение в типизированный носитель | [`src/bot/routers/events.py`](../../src/bot/routers/events.py), [`src/bot/keyboards/__init__.py`](../../src/bot/keyboards/__init__.py) |
| 7 | Двухмашинный workflow через Drive проверен; pre-task cleanup PR на машине-исполнителе может быть пустым | Принуждать каждую машину к pre-task cleanup независимо от состояния working tree | Pre-task cleanup PR — это **способ синхронизации** правок cowork в текущей рабочей копии. Если копия чистая (как было на удалённой машине), коммитить нечего. Ритуал по `CLAUDE.md` остаётся (проверка `git status` обязательна), но если выводит «working tree clean» — переход к feature-ветке корректен | [`handoff/outbox/TASK-014-report.md`](../../handoff/outbox/TASK-014-report.md), [`CLAUDE.md`](../../CLAUDE.md) |
