# ADR 0002 — Monorepo: bot и admin в одном репозитории

**Status:** Accepted
**Date:** 2026-05-22
**Deciders:** cowork-agent, owner

## Context

Бот и админка работают с одной БД и одной доменной моделью (категории, события, исходы, прогнозы, пользователи). Возможные варианты организации репозиториев:

1. Монорепо: один git-репозиторий, общий пакет `shared`, два запускаемых модуля (`bot`, `admin`).
2. Полирепо: отдельные репо для бота, админки и общего пакета (опубликованного как библиотека).
3. Один репо, но без явного `shared` — каждый сервис тащит свою копию моделей.

## Decision

**Монорепо с общим пакетом `src/shared/`** (вариант 1).

Состав:

- `src/bot/` — aiogram-приложение.
- `src/admin/` — FastAPI-приложение.
- `src/shared/` — модели, репозитории, сервисы, конфиг, внешние клиенты, логирование. Импортируется обоими сервисами.
- `src/migrations/` — единый Alembic для всей БД.
- Один `pyproject.toml`, один lock-файл, общий CI.

Два Docker-образа собираются из одного контекста с разными entrypoint.

## Alternatives considered

- **Полирепо** даёт строгую изоляцию, но: придётся поднимать инфраструктуру для приватного PyPI/registry; любая правка модели — это релиз shared → bump в двух репо → синхронизация; для команды из двух агентов это огромный накладной расход.
- **Один репо без shared** — копипаст моделей. Гарантированный путь к рассинхронизации правил («в боте поле обязательно, в админке — нет»), это уже архитектурная ошибка.

## Consequences

**Положительные:**

- Изменения модели/сервисов делаются одним PR, оба сервиса всегда консистентны.
- Один CI, один lock-файл, одна команда `docker compose up`.
- Тесты, фабрики, конфиг — общие.

**Отрицательные / риски:**

- Сильнее связаны жизненные циклы релизов (нельзя сильно отставать в одном сервисе от другого). Митигация: оба сервиса — один продукт, релизятся вместе.
- Базовый образ Docker немного избыточен (в bot-образе есть код admin и наоборот), но это <10MB разницы; разделять multi-stage — преждевременная оптимизация.
