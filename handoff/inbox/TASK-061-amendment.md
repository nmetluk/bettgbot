# Amendment к TASK-061: рассылки приняты частично — 3 дефекта + handoff не закрыт

Код фичи смёржен в `main` (PR #119, `25d2f6b`) и архитектурно **в основном верный** — спасибо:
развязка через БД + scheduler, `FOR UPDATE SKIP LOCKED` в `claim_next_queued`, идемпотентный
`record_delivery` (`on_conflict_do_nothing`), `record` ДО `send_message`, `parse_mode=None`
(плоский текст), пейсинг 0.05s, гард пустого сегмента, CHECK `segment=category ⇒ category_id NOT NULL`,
audit в `enqueue`. Но задача **не закрыта** — три дефекта и незавершённый handoff. Исправить в
ветке `fix/TASK-061-amendment` (или продолжить в исходной), один PR, зелёный CI, затем закрыть по протоколу.

## 🔴 Дефект 1 — ноль тестов (нарушение DoD)

DoD TASK-061 явно требовал integration + unit. «CI 11/11» — это зелёные джобы на 405 **существующих**
тестах; рассылки не покрыты вообще.

Требуется (как в DoD):

- [ ] Integration: `recipients_for` для всех 3 сегментов (фикстуры с blocked / active(last_seen<30д) /
      inactive юзерами и прогнозами в разных категориях — проверить, что `all`/`active`/`category`
      возвращают правильные множества и исключают `is_blocked`).
- [ ] Integration: `claim_next_queued` — берёт самую раннюю `queued`, переводит в `sending`; повторный
      вызов не выдаёт ту же (через delivery-log/статус); корректно возвращает `None` когда очереди нет.
- [ ] Integration: идемпотентность — повторный `record_delivery(b_id, u_id)` возвращает `False` и не дублирует.
- [ ] Integration: `dispatch_broadcasts` со замоканным `bot.send_message` (часть вызовов бросает

      ломает рассылку остальным.
- [ ] Integration: миграция `0005` применяется (расширить `test_migrations` — цепочка `…→0004→0005`).
- [ ] Unit на хэндлеры: `GET /broadcasts` (200, история, пустое состояние), `GET /broadcasts/new` (200),

## 🔴 Дефект 2 — CSP-регресс: инлайн-`<script>` в форме

`src/admin/templates/broadcasts/form.html:99` — инлайн `<script>` (toggle видимости выбора категории
по сегменту). Под текущим CSP (`script-src 'self' https://cdn.jsdelivr.net`, без `unsafe-inline`)
браузер его **заблокирует** → на проде блок «категория» не будет показываться/скрываться, и при
`segment=category` админ не сможет выбрать категорию. Это прямое нарушение DoD TASK-061
(«инлайн-`<script>` запрещён, по паттерну TASK-060») и регресс фикса, сделанного две задачи назад.

- [ ] Вынести логику в `src/admin/static/js/broadcasts.js` (грузится с `'self'`, `defer`);
      инлайн-`<script>` убрать. `git grep '<script>' src/admin/templates/` → пусто.
- [ ] Если JS нужны данные со страницы (id категорий и т.п.) — через `<script type="application/json">`
      или `data-*`-атрибуты (как в `analytics.js`). CSP-заголовок **не менять**.
- [ ] Проверить в браузере: переключение сегмента показывает/прячет dropdown категории; ноль CSP-violation'ов
      в devtools-консоли (описать в отчёте).

## 🔴 Дефект 3 — рассылка не переживает рестарт (идемпотентность не достигнута)

`dispatch_broadcasts` коммитит **один раз в самом конце** (`src/bot/scheduler/jobs.py:203`), а всю
многоминутную рассылку держит в **одной транзакции** с `FOR UPDATE`-локом на строке `broadcast`.
Последствия:
1. Если бот упадёт/перезапустится в середине (отправив, скажем, 5000 из 10000) — транзакция
   откатится: записи `broadcast_delivery` **пропадут**, статус вернётся в `queued`, счётчики обнулятся.
   На следующем тике `claim_next_queued` заберёт её снова и **разошлёт всем повторно** → 5000 уже
   получивших получат **дубликат**. Идемпотентность через delivery-log **не работает**, т.к. записи
   не были закоммичены — ровно та цель из DoD («рестарт в середине не дублирует») не достигнута.
2. Лок `FOR UPDATE` держится всю рассылку — долгая транзакция, удержание блокировки, рост WAL.

Требуется:

- [ ] Закоммитить переход `queued→sending` (+`started_at`) **сразу после claim**, чтобы освободить лок
      и зафиксировать факт старта (не держать `FOR UPDATE` на всё время отправки).
- [ ] Коммитить записи доставки + счётчики **порциями** в цикле (например, каждые ~50 сообщений и в
      конце), чтобы при крэше уже отправленные были зафиксированы.
- [ ] Обеспечить корректное **возобновление** прерванной рассылки: либо `claim` подхватывает «зависшие»
      `sending` (по `started_at` старше N минут) и дорассылает — delivery-log отсеивает уже доставленных;
      либо, если возобновление оставляем на ручной requeue, — **явно задокументировать** это (в docstring
      job'а и отчёте) как осознанное ограничение. Главное требование: **повторов сообщений быть не должно**.
- [ ] Тест на это: записать часть доставок, «прервать», повторно вызвать `dispatch_broadcasts` —
      убедиться, что уже доставленным повторно не шлётся.
