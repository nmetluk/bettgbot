# Решения — task-013-review

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | `assert user is not None` в handler'ах после `@require_active_user` остаётся | `cast(User, user)` / явный тип `user: User` в сигнатуре с `# type: ignore` | `assert` даёт runtime-safety (если декоратор по ошибке снят — handler упадёт явным AssertionError, а не обнаружит проблему через NoneType deref глубже в коде). `cast` подавляет mypy без runtime-проверки — хуже. Изменение типа в сигнатуре handler'а ломает совместимость с aiogram DI (он передаёт `user` из middleware как `User \| None`, и фильтр на сигнатуру не накладывается) | [`state/DECISIONS.md`](../../state/DECISIONS.md), [`src/bot/routers/events.py`](../../src/bot/routers/events.py) |
| 2 | Fallback handlers `on_predict_pick_no_state` / `on_predict_confirm_no_state` отвечают alert «Событие больше недоступно» | `query.message.edit_text(...) + кнопка "Открыть событие заново"` | Alert — минимально достаточный UX: пользователь нажал кнопку из устаревшего сообщения, бот честно сказал «состояние потеряно», пользователь сам открывает «📅 Все события». Edit_text+кнопка добавляет ещё один entrypoint и требует знания event_id (он есть в callback_data, но всё же). Преждевременное усложнение | [`state/DECISIONS.md`](../../state/DECISIONS.md), [`src/bot/routers/prediction.py`](../../src/bot/routers/prediction.py) |
| 3 | `render_event_card` остаётся в `src/bot/routers/events.py` с экспортом через `__all__` | Вынести в `src/bot/_event_card.py` / `src/bot/views.py` | Правило тройки: одно дополнительное использование (`prediction.py → on_predict_cancel`) не оправдывает создание нового модуля. Логически карточка события принадлежит events-flow. Локальный импорт в `prediction.py` — одна строка, без circular. Триггер выноса — третье место использования (TASK-014 будет звать тот же `render_event_card`, но через `events.py` — это всё ещё «один экспорт», как сейчас, и точкой использования становится `my.py`. Это третья точка, но синтаксически тот же импорт. Перевешу к выносу, когда появится **четвёртая** точка) | [`state/DECISIONS.md`](../../state/DECISIONS.md), [`src/bot/routers/events.py`](../../src/bot/routers/events.py) |
| 4 | `on_predict_cancel` без state-фильтра — design choice, не баг консистентности | Добавить state-фильтр и продублировать `on_predict_cancel_no_state` с тем же поведением | Отмена — глобальное действие пользователя. Если он жмёт «❌ Отмена» из устаревшего FSM-сообщения (после рестарта бота, истечения FSM), бот корректно идёт по нормальному пути: `state.clear()` + рендер карточки. Не нужно делать fallback с alert «состояние потеряно» — потому что нет состояния, которое мы могли бы потерять для отмены | [`state/DECISIONS.md`](../../state/DECISIONS.md), [`src/bot/routers/prediction.py`](../../src/bot/routers/prediction.py) |
| 5 | `@require_active_user` в `cmd_predict → cmd_events` срабатывает дважды — оставляем | `cmd_predict` зовёт `cmd_events.__wrapped__` (минуя декоратор второй раз) | Двойная проверка — два `if`-сравнения на in-memory объекте, без SQL (user уже в data из middleware). Накладной расход незаметный. Использование `__wrapped__` опирается на скрытое поведение `functools.wraps` и легко сломается при следующем рефакторинге декоратора. YAGNI | [`state/DECISIONS.md`](../../state/DECISIONS.md), [`src/bot/routers/prediction.py`](../../src/bot/routers/prediction.py) |
