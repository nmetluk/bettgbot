# ADR 0006 — Открытая регистрация (отмена проверки по внешнему реестру)

**Status:** Accepted
**Date:** 2026-06-01
**Deciders:** owner (Николай), cowork-agent
**Supersedes:** часть [ADR 0001](0001-tech-stack.md) (проверка пользователей через внешний реестр), [docs/06-external-api.md](../06-external-api.md)

## Context

Изначально (TASK-008, TASK-009, [docs/06-external-api.md](../06-external-api.md)) регистрация в боте была gated: бот принимал контакт, нормализовал телефон до E.164 и вызывал `ExternalUserRegistryClient.verify(phone)`. Пользователь создавался только если внешний реестр отвечал «разрешён». Под это была построена инфраструктура:

- модуль `src/shared/external/` — `Protocol` `ExternalUserRegistryClient`, HTTP-реализация, mock на YAML/env;
- фабрика `get_registry_client()` + DI-внедрение в `dispatcher["registry"]`;
- исключения `UserNotAllowed`, `RegistryUnavailableError`, `ExternalApiError`;
- тексты бота `PHONE_NOT_FOUND`, `REGISTRY_UNAVAILABLE`;
- конфиг `ExternalRegistrySettings` (`EXTERNAL_REGISTRY_BACKEND`, `EXTERNAL_API_*`, `MOCK_REGISTRY_*`) и prod-валидация «mock не допускается в prod».

**2026-06-01 заказчик изменил требование:** проверка по внешней базе перед регистрацией больше не нужна. Любой пользователь может зарегистрироваться, отправив свой контакт.

## Decision

1. **Регистрация становится открытой.** `UserService.register_or_authenticate` создаёт пользователя сразу после получения контакта, без вызова внешнего API. Единственный отказной путь — `is_blocked = true` (блокировка из админки).
2. **Внешний реестр удаляется полностью** (выбор владельца из вариантов «удалить / отключить / спрятать за флагом»): модуль `src/shared/external/`, фабрика и DI-проводка, исключения реестра, тексты, конфиг и env-переменные, связанные тесты, mock-фикстуры.
3. **Схема БД не меняется.** `external_user_id`/`display_name` из реестра никогда не сохранялись в `users`, миграции не нужны.
4. **`docs/06-external-api.md` помечается SUPERSEDED** и сохраняется как историческая справка на случай, если проверку вернут.

## Consequences

**Плюсы:** проще флоу регистрации и кодовая база; нет внешней зависимости и связанного класса ошибок; не нужно согласовывать контракт с владельцем внешней системы; убирается prod-блокер «нужен `EXTERNAL_REGISTRY_BACKEND=http`».

**Минусы / риски:** регистрироваться может кто угодно — нет гейта на входе. Это осознанный выбор заказчика; модерация при необходимости делается реактивно через блокировку (`is_blocked`) в админке. Если бизнес-требование вернётся — восстановление по git-истории удалённого модуля и снятому SUPERSEDED-баннеру 06.

**Отменяемые ранее принятые решения** (DECISIONS.md, теперь неактуальны как руководство к действию): связанные с реестром записи от 2026-05-23 — `get_registry_client()` singleton, `X-Request-Id` на ретраи, `_phone_hash`-дублирование, `RegistryUnavailableError` как warning, `StubRegistry`, `load_mock_config` defensive, `fail_rate`, type-only импорты из подмодуля external. Историческая ценность сохраняется, но код, к которому они относятся, удаляется.
