# Глоссарий

Единые термины предметной области и проекта. Использовать **только** их — и в коде (имена сущностей, методов), и в документации, и в UI-текстах.

## Предметная область

| Термин | EN | Определение |
|---|---|---|
| **Категория** | `Category` | Логическая группа («папка») событий. Например: «Футбол», «Хоккей», «Киберспорт», «Прочее». |
| **Событие** | `Event` | Объект, на исход которого пользователь делает прогноз. Имеет название, описание, дату/время начала, дедлайн приёма прогнозов, категорию и набор возможных исходов. |
| **Исход** | `Outcome` | Возможный вариант результата события (например, для матча: «победа команды A», «ничья», «победа команды B»). |
| **Прогноз** | `Prediction` | Выбор пользователем конкретного исхода для события. Один пользователь — один прогноз на событие. |
| **Итог** | `Result` (поле `Event.result_outcome_id`) | Фактический исход события после его завершения. Заносится администратором. |
| **Архив** | `archive` (флаг `Event.is_archived`) | Состояние события после фиксации итога: оно перестаёт показываться пользователям в активных списках, но видно в «Мои прогнозы → Архив» и в админке. |
| **Сбылся / не сбылся** | `is_correct` (поле `Prediction.is_correct`) | Признак, что выбранный пользователем исход совпал с зафиксированным итогом события. Заполняется автоматически при фиксации `result_outcome_id`. |
| **Дедлайн приёма прогнозов** | `Event.predictions_close_at` | Момент времени, после которого новые прогнозы не принимаются. Обычно равен или раньше начала события. |
| **Напоминание** | `Reminder` | Уведомление пользователю в Telegram за заданное время до события — если у него ещё нет прогноза по этому событию. |
| **Настройки напоминаний** | `ReminderSetting` | Per-user конфигурация: глобально включены/выключены, временные офсеты (например «за 24 часа», «за 1 час»). |

## Роли

| Термин | EN | Определение |
|---|---|---|
| **Пользователь** | `User` | Конечный пользователь Telegram, прошедший проверку по номеру телефона через внешний реестр. |
| **Администратор** | `AdminUser` | Сотрудник, имеющий доступ в веб-админку. Учётка отдельна от Telegram-пользователя. |

## Внешние системы

| Термин | EN | Определение |
|---|---|---|
| **Внешний реестр пользователей** | `External User Registry` | Сторонняя система, в которой ведётся справочник разрешённых телефонов. Проверка через HTTP API. На этапе разработки — mock. |

## Процесс / инфраструктура

| Термин | EN | Определение |
|---|---|---|
| **Handoff** | `handoff` | Папка обмена задачами и отчётами между cowork-агентом и локальным Claude Code. Не «inbox/outbox в почте», а именно проектный канал — см. [`handoff/README.md`](../handoff/README.md). |
| **Сессия проектирования** | `design session` | Один рабочий заход cowork-агента, оставляющий след в `sessions/YYYY-MM-DD-NN-slug/`. |
| **TASK-NNN** | — | Идентификатор задачи в handoff. Монотонный, не переиспользуется. |
| **ADR** | `Architecture Decision Record` | Документ с обоснованием крупного архитектурного решения. Лежит в `docs/adr/`. |
| **MVP** | — | Минимальный набор фич, после которого бот может быть запущен в продуктив: регистрация, CRUD событий, приём прогнозов, фиксация итога, статистика, напоминания, аудит. |

## Использование

- В коде: классы в PascalCase из колонки EN (`Event`, `Prediction`), атрибуты — `snake_case`.
- В UI-текстах и документации: всегда RU-термин из колонки «Термин». Никаких синонимов («ставка», «прогноз» — выбран «прогноз»; «турнир», «событие» — выбрано «событие»).
- В коммитах и PR: можно использовать оба, но при первом упоминании в описании — оба варианта (`feat(events): add archive flag (архивация)`).
