Если внешний API использует ID для идемпотентности (типичный паттерн дедупликации) — повторы с тем же ID помогут детектировать дубли. Различение попыток уже даёт `retry_count` в логах. Уровни correlation у нас: middleware → bot/web → external (`X-Request-Id`); каждый — своя ответственность | [`state/DECISIONS.md`](../../state/DECISIONS.md), TASK-009 Step 0 |
| 2 | `get_registry_client()` — singleton через `@lru_cache(maxsize=1)` | На каждый вызов новый client (текущая реализация в TASK-008) | HTTP connection pool теряет смысл без singleton; новый `httpx.AsyncClient` каждый вызов — пустая трата ресурсов. Тесты делают `cache_clear()` как с `get_settings()`. Шатдаун консьюмеры дёрнут `await client.close()` (TASK-010 main.py) | [`state/DECISIONS.md`](../../state/DECISIONS.md), TASK-009 Step 0 |

Молчаливый no-op обнаружится в первом же `verify()`. Pydantic-валидатор добавил бы код, который придётся поддерживать без выигрыша | [`state/DECISIONS.md`](../../state/DECISIONS.md) |
| 4 | `fail_rate` использует глобальный `random` (не seeded `Random` инстанс) | Параметр `Random` в конструкторе для воспроизводимости | Тесты используют 0.0/1.0 — детерминированные крайности. Сценарий «30% fail воспроизводимо» сейчас не требуется; добавим параметр, когда понадобится | [`state/DECISIONS.md`](../../state/DECISIONS.md) |
