# ADR 0005 — Стек веб-админки v2

**Status:** Proposed
**Date:** 2026-05-29
**Deciders:** cowork-agent, owner (Николай) — решение ожидается
**Supersedes (частично):** строку «React/Vue SPA для админки» в [ADR 0001](0001-tech-stack.md), если будет принят вариант B

## Context

MVP-админка спроектирована и реализована как SSR на FastAPI + Jinja2 + HTMX + Bootstrap 5
(см. [ADR 0001](0001-tech-stack.md), [docs/05-admin-spec.md](../05-admin-spec.md)). В ADR 0001
вариант «React/Vue SPA для админки» был сознательно отвергнут: «лишний build-pipeline, никаких
сильных требований к интерактивности; HTMX закрывает 95% сценариев минимальной админки».

2026-05-29 дизайнеры передали интерактивный прототип админки **для следующей версии**
([`sessions/2026-05-29-01-admin-design/`](../../sessions/2026-05-29-01-admin-design/)). Прототип
построен как **клиентский SPA на React 18** (Babel standalone, без сборки) с **собственной
дизайн-системой** на CSS-токенах (`--pv-*`): светлая/тёмная тема, плотность таблиц, настраиваемый
акцент, расширенный набор экранов (дашборд, категории, события, карточка события с фиксацией
результата, пользователи, аудит-лог, каталог пост-MVP-идей).

Это прямо расходится с принципами действующей спецификации («никакого SPA», «никаких собственных
дизайн-токенов»). Нужно решить, как развивать админку в v2, не ломая то, что уже работает, и не
обесценивая переданную дизайнерами работу.

Контекст, который не изменился со времён ADR 0001:

- админка — внутренний инструмент с низкой нагрузкой (события создаются ≤1/день, пользователей — тысячи, не миллионы);
- команда — два агента + владелец, без выделенного фронтенд-разработчика;
- реализация идёт через handoff-задачи; чем меньше движущихся частей, тем дешевле сопровождение.

Что изменилось:

- появилась проработанная дизайн-система и набор экранов, которых раньше не было;
- объём желаемого интерфейса вырос (аудит-лог, дашборд со сводками, roadmap-экраны).

## Decision

**Решение пока не принято (Status: Proposed).** Ниже — варианты и рекомендация архитектора.
Финальный выбор — за владельцем; после фиксации этот ADR переводится в `Accepted`, а
соответствующие части [docs/05-admin-spec.md](../05-admin-spec.md) и [ADR 0001](0001-tech-stack.md)
обновляются.

### Рассмотренные варианты

**Вариант A — остаться на SSR, прототип как визуальный референс.**
Архитектура без изменений (Jinja2 + HTMX + Bootstrap 5). Из прототипа берём только визуальный язык
по мере необходимости. Прототип хранится как референс, в продакшен не идёт.
- ➕ Ноль изменений в стеке и инструментах; полностью совместимо с ADR 0001/0004 и уже написанным кодом.
- ➕ Минимум работы для исполнителя; нет build-pipeline, нет раздельного API.
- ➖ Богатый дизайн (тёмная тема, токены, плотность, интерактивные экраны) воспроизводится лишь частично; работа дизайнеров используется не полностью.

**Вариант B — полный переход на React SPA (как в прототипе).**
Админка становится SPA; бэкенд FastAPI отдаёт JSON-API вместо HTML; добавляется фронтенд-сборка
(Vite/esbuild), отдельный пайплайн линта/тестов фронтенда.
- ➕ Дизайн переносится 1:1, вся дизайн-система и интерактив сохраняются.
- ➕ Потолок по интерактивности выше (пригодится для аналитики/лидербордов из roadmap-экранов).
- ➖ Реверс ключевого решения ADR 0001: новый build-pipeline, раздельные API/клиент, удвоение
  поверхности тестирования и CI, рост сложности handoff-задач.
- ➖ Нет выделенного фронтенд-разработчика; сопровождение SPA силами двух агентов дороже.
- ➖ Прототип использует Babel-in-browser (CDN, без сборки) — это годится для демо, но не для
  продакшена; потребуется полноценная пересборка проекта на нормальный bundler.

**Вариант C (рекомендуется) — SSR-архитектура + порт дизайн-системы.**
Сохраняем SSR + HTMX (ADR 0001/0004), но **принимаем визуальный язык прототипа как дизайн-спеку**:
переносим токены `--pv-*`, сетку, тёмную тему и плотность в собственный CSS поверх (или вместо)
Bootstrap-темы; экраны реализуем в Jinja-шаблонах, точечную интерактивность — на HTMX (+ Alpine.js
для мелких клиентских состояний вроде переключателя темы/плотности). React-прототип остаётся
интерактивным эталоном «как должно выглядеть и вести себя».
- ➕ Дизайн дизайнеров используется почти полностью (визуал, токены, тёмная тема, плотность).
- ➕ Без реверса ADR 0001: нет SPA, нет раздельного API, переиспользуется существующий код роутов/сервисов.
- ➕ Build-pipeline остаётся минимальным (CSS + опционально Alpine из CDN/локально).
- ➖ Часть «дорогой» интерактивности прототипа придётся упростить или реализовать на HTMX/Alpine, а не на React-состоянии.
- ➖ Пункт «никаких собственных дизайн-токенов» из спеки отменяется — это осознанное изменение, фиксируется этим ADR.

### Рекомендация

**Вариант C.** Он максимально сохраняет работу дизайнеров (визуальный язык, токены, тёмная тема),
не разрушая архитектурных решений, на которых уже построен MVP, и не навешивая на двухагентную
команду стоимость сопровождения SPA. Вариант B оправдан только если в roadmap появятся экраны с
действительно высокой клиентской интерактивностью (живые дашборды, drag-and-drop), которые HTMX
закрывает плохо — тогда стоит вернуться к этому ADR и пересмотреть.

## Consequences

**Если принят вариант C:**

- В [docs/05-admin-spec.md](../05-admin-spec.md) принцип «никаких собственных дизайн-токенов»
  заменяется на «дизайн-токены из прототипа v2»; принцип SSR/HTMX сохраняется.
- Заводится handoff-задача: порт дизайн-системы (`styles.css` → CSS-токены проекта + Bootstrap-overrides),
  затем поэкранный перенос в Jinja-шаблоны.
- Опционально добавляется Alpine.js (зафиксировать в [docs/02-tech-stack.md](../02-tech-stack.md))
  для клиентских состояний без выхода на полноценный SPA.
- React-прототип остаётся в `sessions/.../artifacts/admin/` как замороженный визуальный эталон.

**Если принят вариант B:**

- Этот ADR переводит в `Superseded` соответствующую строку ADR 0001; заводится новый ADR на
  архитектуру SPA + JSON-API; [docs/02-tech-stack.md](../02-tech-stack.md) и
  [docs/05-admin-spec.md](../05-admin-spec.md) переписываются под клиент-серверное разделение;
  в CI добавляется фронтенд-пайплайн.

**Общее:**

- До принятия решения новые задачи на реализацию v2-админки **не заводятся** — иначе рискуем
  написать код под стек, который потом сменится.
