# ADR 0001 — Технологический стек

**Status:** Accepted
**Date:** 2026-05-22
**Deciders:** cowork-agent, owner (Николай)

## Context

Нужно выбрать стек для Telegram-бота с веб-админкой. Ожидаемый объём — несколько тысяч пользователей, события создаются редко (≤1 в день). Биллинга нет. Дизайнерских ресурсов нет — для админки нужна готовая тема. Деплой на один VPS. Команда — два агента (проектировщик + исполнитель) + один владелец.

Требования:

- одна кодовая база, единая бизнес-логика для бота и админки;
- хорошая поддержка async (Telegram + веб-роуты);
- зрелые инструменты миграций, тестирования, линтинга;
- транзакционность для критичной операции «фиксация итога + отметка прогнозов»;
- готовность к расширению (статистика, многоязычность);
- минимум boilerplate.

## Decision

Принятый стек:

- **Python 3.12** + **aiogram 3.x** (бот) + **FastAPI** (админка).
- **SQLAlchemy 2.0 (async) + Alembic + PostgreSQL 16**.
- **Redis 7** для FSM-storage и кеша.
- **Jinja2 + HTMX + Bootstrap 5** (готовый Bootstrap-шаблон).
- **httpx**, **structlog**, **APScheduler**, **passlib[bcrypt]**, **pydantic-settings v2**.
- **pytest + pytest-asyncio + factory-boy + freezegun**.
- **ruff** (lint+format), **mypy** (strict для shared/).
- **Docker Compose** для дев и прод, **GitHub Actions** для CI.

Полная таблица — в [`../02-tech-stack.md`](../02-tech-stack.md).

## Alternatives considered

| Альтернатива | Почему отвергнуто |
|---|---|
| **python-telegram-bot** вместо aiogram | Зрелый, но aiogram 3 удобнее для async-first, FSM из коробки, более активная экосистема |
| **Django + Django Admin** | Дублирование стека (sync ORM рядом с async aiogram); Django Admin удобен, но монолитен и навязывает свой стиль; для минимальной админки переплата сложностью |
| **SQLite вместо PostgreSQL** | Допустимо для proto, но для нескольких тысяч пользователей и админских join/aggregate-запросов PostgreSQL даёт лучший потолок; миграция в будущем — дороже, чем сразу взять Postgres |
| **Node.js (grammY + Next.js)** | Современный JS-стек, но владелец не выразил предпочтение; Python экосистема даёт более ровный стек именно для серверной части без UI-кастомизации |
| **Flask + Flask-Admin** | Зрелый, но sync и менее удобный async-интерой, чем FastAPI; меньше DI |
| **React/Vue SPA для админки** | Лишний build-pipeline, никаких сильных требований к интерактивности; HTMX закрывает 95% сценариев минимальной админки |
| **Celery для фоновых задач** | Избыточно для текущих объёмов; APScheduler внутри bot-процесса проще |

## Consequences

**Положительные:**

- Один язык, одна ORM, общая бизнес-логика для бота и админки — меньше дублирования, проще тестировать.
- Async-стек по всей вертикали — однородный профиль производительности.
- Docker Compose даёт воспроизводимый дев и прод одной командой.
- Готовый Bootstrap-шаблон закрывает админку без дизайнерской работы.

**Отрицательные / риски:**

- aiogram 3 — относительно молодая major-версия; есть риск breaking-changes в minor (митигируем фиксацией версии в lock-файле).
- SQLAlchemy 2.0 async ORM требует от исполнителя дисциплины (правильные `async with session.begin()`, никаких lazy-загрузок вне сессии) — закрывается code review и линтером.
- APScheduler внутри бота — не масштабируется на несколько инстансов; если в будущем понадобится — миграция на Celery/external scheduler (планируем как backlog-задачу, не блокер для MVP).
