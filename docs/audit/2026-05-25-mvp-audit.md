## Executive Summary

### Состояние проекта одним абзацем

За четыре дня создан **технически добротный MVP**: 7,5 тыс. строк кода в `src/`, 6,9 тыс. — в тестах (343 теста), полная админка с CSRF и signed-cookie сессиями, два фоновых job'а, три миграции, развёрнутый Docker-compose (dev + prod + nginx + certbot + pg_dump-бэкапы), отдельный CI-pipeline. Соблюдены конвенции (handler→service→repo→model), доменные исключения с `Literal["reason"]`, structlog, mypy `--strict` для shared-слоя, async-PostgreSQL через asyncpg. Двухагентный workflow с `handoff/`-протоколом отработан без сбоев. **Это сильный фундамент.**

При этом до полноценного production-deploy проект **не готов**. Найдено **10 critical-блокеров**, **24 high-проблемы**, **30 medium** и **28 low/info** замечаний. Из них:
- **5 уязвимостей безопасности класса CWE-639/79/352/770** в коде админки и бота;
- **отсутствует offsite-бэкап БД и DR-план** — физическая потеря VPS = потеря всех данных;
- **нет внешнего monitoring/alerting** — первый production-инцидент найдут пользователи, не операторы;
- **нет security-сканирования** (bandit, pip-audit, trivy, gitleaks, Dependabot);
- **дефолтные dev-секреты** в `.env` без валидации, что они не утекут в prod.

### Что в проекте сделано хорошо (валидация сильных сторон)

1. **Архитектурная дисциплина.** Слоистая модель handler → service → repository → model в основном соблюдается, exceptions для бизнес-логики, `Literal["reason"]` для exhaustive `match`, decision-журнал ведётся как живой документ.
2. **Тесты.** 343 теста (227 unit + 116 integration), realistic Postgres fixtures с SAVEPOINT, миграции тестируются `subprocess uv run alembic` (то же, что в prod).
3. **CI зелёный** на 5 job'ах (lint, typecheck, test, integration, handoff-consistency).
4. **Security-практики, которые сделаны правильно**: bcrypt напрямую (не passlib), timing-attack mitigation в auth, CSRF + rate-limit на /login, anti-enumeration на одинаковом 401-сообщении, sha256-маска телефона в логах, signed-cookie через `itsdangerous`, `Secure`-flag conditional на `ENVIRONMENT != dev`, `expire_on_commit=False` для async, циклическая FK через `use_alter`, partial-index `ix_event_predictions_close_at_active`, UTC везде через `datetime.now(tz=UTC)`.
5. **Идемпотентность scheduler-job** через `record → send_message` и UNIQUE constraint.
6. **Документация**: 10 spec-документов в `docs/`, ADR-журнал, GLOSSARY, BACKLOG, PROJECT_STATUS — лучше, чем в большинстве стартапов.
7. **Docker multistage** + non-root user `bb`, healthchecks на всех сервисах, `depends_on: condition: service_healthy`.

## CRITICAL (10 находок) — блокеры прод-деплоя

### C-01. IDOR в outcomes-CRUD: `event_id` из URL не валидируется относительно `outcome.event_id`
**Severity:** Critical · **CWE-639** · **Файл:** `src/admin/routes/outcomes.py:132-179`, `src/shared/services/event.py:182-201`

`POST /events/{event_id}/outcomes/{outcome_id}` и `POST /events/{event_id}/outcomes/{outcome_id}/delete` передают оба параметра, но сервис фильтрует **только по `outcome_id`**:

```python
# src/admin/routes/outcomes.py:146
await EventService(session).update_outcome(
    outcome_id=outcome_id,  # event_id не передаётся
    ...
)
```

Аутентифицированный admin может перейти на `/events/1/outcomes/999` и мутировать outcome `999`, который принадлежит другому событию. Это нарушает атомарность инварианта Event ↔ Outcome (если когда-то появятся per-event роли, IDOR станет privilege escalation).

**Фикс:** в `EventService.update_outcome`/`delete_outcome` обязательным параметром добавить `event_id`, в `update(...).where(...)` добавить `Outcome.event_id == event_id`, при 0 rowcount поднимать `OutcomeNotForEventError`.

### C-02. Дефолтные dev-секреты в `.env` без prod-валидации
**Severity:** Critical · **CWE-798, CWE-521** · **Файл:** `.env:6-47`, `src/shared/config.py:117-145`

```
.env:6  TELEGRAM_BOT_TOKEN=dev-bot-token
.env:10 POSTGRES_PASSWORD=changeme
.env:20 ADMIN_SECRET_KEY=dev-admin-secret
.env:47 ADMIN_CSRF_SECRET=dev-csrf-secret
```

`.env` в `.gitignore` — не утечёт в репо, ок. Но если оператор скопирует dev-`.env` на prod-VPS (типичный сценарий), `Settings` примет `dev-admin-secret` без замечаний. Этот ключ — base подписи cookie сессии админа: forging тривиально (`itsdangerous.dumps({"admin_id": 1})` с этим секретом → готовая cookie). Захват админки за минуту.

**Фикс:** `@model_validator(mode="after")` в `Settings`: если `environment != "dev"` → запретить `secret_key`/`csrf_secret` равные dev-значениям, короче 32 символов, или совпадающие. Сгенерировать prod-секреты в deploy-runbook через `python -c "import secrets; print(secrets.token_urlsafe(64))"`.

Связанная проблема: `mock_registry` тоже не запрещён в prod — `EXTERNAL_REGISTRY_BACKEND=mock` в prod значит, что регистрация работает по списку в `infra/mock-registry.yml` (любой может зарегистрироваться, если подделать его телефон в файле). Тот же валидатор должен запретить `mock` в `environment != "dev"`.

### C-03. CSRF cookie не ротируется → effectively static per session 8 часов
**Severity:** Critical · **CWE-352** · **Файл:** `src/admin/auth/middleware.py:149`, `src/admin/routes/outcomes.py:108,154,179`, `src/admin/routes/login.py:107`

`CsrfTokenMiddleware` генерирует токен **только на GET-запросах**:
```python
# src/admin/auth/middleware.py:149
if request.method != "GET" or path.startswith("/static/") or path == "/healthz":
    await self.app(scope, receive, send)
    return
```

HTMX-handler'ы для outcomes возвращают рендеренные fragment-ы прямо из POST (`_list_response` без перегенерации). Эти фрагменты содержат `{{ request.state.csrf_token }}`, который **никогда не выставлен** на POST → ssh-attribute пустая. Дальнейшие HTMX POST'ы шлют пустой `csrf_token` в body, валидируясь против всё ещё валидной `fastapi-csrf-token` cookie. Cookie живёт всю сессию (8 часов sliding-TTL у session-cookie), и эта же CSRF-cookie shareable между админами на одном host. Login также не регенерирует CSRF-cookie. Окно атаки = 8 часов фиксированного токена.

**Фикс:** (а) в HTMX-handler'ах outcomes/categories/events/users явно вызывать `csrf_protect.generate_csrf_tokens()` + `csrf_protect.set_csrf_cookie(...)` (паттерн уже реализован в `_render_login_error` и `_render_form_with_error` — обобщить). (б) на `POST /login` после успеха перевыпускать CSRF-cookie. (в) Использовать `SameSite=Strict` для admin-only UI (см. M-05).

### C-04. HTML injection в Telegram-сообщениях через admin-controlled поля
**Severity:** Critical · **CWE-79** · **Файл:** `src/bot/main.py:42-43`, `src/bot/texts.py:105-114`, `src/bot/routers/events.py:158-166`

Бот глобально установлен в `ParseMode.HTML`:
```python
# src/bot/main.py:42-43
bot = Bot(token=..., default=DefaultBotProperties(parse_mode=ParseMode.HTML))
```

Все шаблоны (`EVENT_CARD`, `MY_ROW_ACTIVE/ARCHIVE`, `PREDICT_*`, `WELCOME_NEW_REGISTERED`, `REMINDER_NOTIFICATION`) собираются через `texts.X.format(title=event.title, label=outcome.label, first_name=user.first_name, ...)` **без `html.escape` или `aiogram.html.quote`**. Admin, который вводит `Match: A vs B<a href="https://evil.com/phish">tap here</a>` в название события, рассылает этот «тег» всем подписчикам напоминаний. Сценарий тривиальный: компрометация одного админа = массовая phishing-рассылка через офф-канал доверенной коммуникации.

Дополнительный риск — DoS: title с непарным `<` валит aiogram в `Bad Request: can't parse entities`, событие пропадает из каталога и сломаны все будущие напоминания по нему.

**Фикс:** на каждом `.format()`-вызове в боте обернуть user/admin-supplied поля в `aiogram.utils.markdown.html_decoration.quote(...)` (или `html.escape(..., quote=False)`). Defence-in-depth — серверная валидация: запретить `<` в форме события (см. M-21).

### C-05. Нет security-headers (CSP, X-Frame, Permissions-Policy) в FastAPI; nginx даёт минимум, CDN-ассеты без SRI
**Severity:** Critical · **CWE-693, CWE-829** · **Файл:** `src/admin/app.py:71-130`, `src/admin/templates/base.html:8-12`, `infra/nginx/admin.conf.template:29-31`

FastAPI не выставляет никаких security-headers. Nginx добавляет HSTS (без `includeSubDomains; preload`), X-Frame-Options DENY и nosniff — но: (а) при доступе вне nginx (`make admin` на dev, или другой reverse-proxy) заголовков нет; (б) **никакой CSP**; (в) Bootstrap 5.3.3 и HTMX 2.0.4 грузятся с `cdn.jsdelivr.net` без `integrity="sha384-..."` и без `crossorigin`. Компрометация jsdelivr или DNS-хайджек admin-машины = выполнение произвольного JS в админке. В сочетании с H-21 (XSS в `onsubmit`/`hx-confirm`) — нет второй линии защиты.

```html
<!-- src/admin/templates/base.html:8 -->
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/css/bootstrap.min.css">
<!-- src/admin/templates/base.html:12 -->
<script src="https://cdn.jsdelivr.net/npm/htmx.org@2.0.4/dist/htmx.min.js"></script>
```

**Фикс:** добавить ASGI-middleware `SecurityHeadersMiddleware` со строгой CSP (`default-src 'self'; script-src 'self' https://cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; img-src 'self' data:; frame-ancestors 'none';`), `X-Frame-Options DENY`, `X-Content-Type-Options nosniff`, `Referrer-Policy strict-origin-when-cross-origin`, `Permissions-Policy "geolocation=(), camera=(), microphone=()"`. Добавить SRI-хэши на CDN-ссылки или (лучше) self-host под `/static/`. Обновить nginx: HSTS с `includeSubDomains; preload`, OCSP stapling, `ssl_session_tickets off`, ciphers из Mozilla Intermediate.

### C-06. Rate-limit и логи берут IP из `request.client.host`, который за nginx = IP контейнера nginx
**Severity:** Critical · **CWE-770, CWE-307** · **Файл:** `src/admin/routes/login.py:67,86`, `infra/Dockerfile.web:37`, `infra/nginx/admin.conf.template:42`

```python
# src/admin/routes/login.py:67
@router.post("/login", dependencies=[Depends(RateLimiter(times=5, seconds=60))])
# src/admin/routes/login.py:86
ip=request.client.host if request.client else None,
```

`fastapi-limiter` дефолтным identifier'ом берёт `request.client.host`. За nginx это **IP nginx-контейнера** (`172.x.x.x` в docker-network). Это значит:
1. **5 failed login/60s `глобально`** — не per-IP. Один атакующий перекрывает rate-limit для всех админов.
2. **Real IP атакующего теряется в `audit.log`** — incident response невозможен.
3. Атака credential-stuffing незаметна для оператора.

Дополнительно: `uvicorn` запущен без `--proxy-headers --forwarded-allow-ips=<nginx-IP>`, поэтому `request.client.host` и Starlette даже не пытаются прочитать `X-Forwarded-For`.

**Фикс:** (а) запустить uvicorn с `--proxy-headers --forwarded-allow-ips="*"` (только за trusted nginx); (б) кастомный identifier для `fastapi-limiter`, который комбинирует IP (из `X-Forwarded-For`, левый-most не-trusted) и `login`-поле; (в) на уровне nginx добавить `limit_req_zone $binary_remote_addr zone=login:10m rate=10r/m` и `limit_req zone=login burst=5`.

### C-07. Нет offsite/encrypted backup БД и DR-плана
**Severity:** Critical · **CWE-693** · **Файл:** `infra/docker-compose.prod.yml:63-91`, `scripts/backup-to-drive.sh:71-94`, `docs/07-deployment.md`

`db-backup` контейнер делает ежедневный `pg_dump | gzip` в named volume `bb-db-backups` **на том же VPS**. `scripts/backup-to-drive.sh` зеркалит только handoff/state/sessions, **БД не копирует**. Если VPS физически потерян (фаер, ransomware, terminate by hosting provider) — все данные потеряны. Шифрования-at-rest нет. Retention 14 дней без проверки восстановимости (никаких smoke-restore тестов). RTO/RPO не определены. DR-runbook («как поднять с нуля на новом VPS») не существует.

**Фикс:** (а) добавить sidecar-контейнер с `rclone` или `restic` для ежедневного шифрованного push дампов в S3/B2/GDrive (`pg_dump ... | age -e -r <pubkey> | rclone rcat ...`); (б) еженедельная проверка `pg_restore --list` против последнего бэкапа; (в) написать `docs/runbook-dr.md` с RTO=8h, RPO=24h, пошаговым восстановлением.

### C-08. Отсутствует security-сканирование в CI (bandit, pip-audit, trivy, gitleaks, Dependabot)

Подключить Dependabot (`.github/dependabot.yml` с weekly cadence для pip + docker + github-actions).

### C-09. Нет внешнего monitoring/alerting (Sentry, Healthchecks.io, Prometheus)
**Severity:** Critical · **CWE-778** · **Файл:** `pyproject.toml`, `src/{bot,admin}/`

Логи через `structlog` JSON — только в локальных Docker-файлах `/var/lib/docker/containers/.../*.log`, ring-buffer 10 MB × 3 (60 MB на сервис, ~6 часов истории на нагруженном проде). Никакого Sentry/Datadog для ошибок. Никакого Prometheus для метрик. Никакого Healthchecks.io / UptimeRobot для liveness. Первый production-инцидент будет обнаружен пользователями через Telegram.

# Замечания о методе

1. Я **не делал runtime-проверок**. Все находки — статический анализ кода + конфигурации. Часть из них может оказаться false-positive в работающей системе (например, если bcrypt 5 действительно не raises для длинных паролей в текущей версии). Перед фиксом каждой critical-проблемы рекомендую воспроизводить (где это безопасно).

2. Я **не запускал тесты**, не строил образы, не разворачивал compose. Доверял PROJECT_STATUS, DECISIONS и handoff/outbox.

3. Subagent-ы привлечены для параллельного сбора находок по трём областям (security / architecture / infra+tests). Их выводы сведены, дедуплицированы и факт-чекнуты против исходников.

4. **Severity-классификация субъективна**: я ориентируюсь на типовые threat-модели публичного веб-сервиса с админкой и Telegram-каналом. Для closed-perimeter VPS (доступ только из VPN) часть critical может быть понижена в P1.

5. Я **не оценивал бизнес-логику** глубоко — её корректность подтверждается 343 тестами и приёмочными циклами. Аудит — про NFR (non-functional requirements).
