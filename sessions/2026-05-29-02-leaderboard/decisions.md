# Решения — leaderboard

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | Метрика ранжирования — точность `correct/resolved*100`, по убыванию | По числу верных; по «очкам» | Точность честнее к разной активности; число верных — вторичный критерий (tiebreak) | `TASK-058` |
| 2 | Порог квалификации — минимум `N` разрешённых прогнозов (дефолт N=5) | Без порога | Без порога 1/1=100% забивает топ; порог отсекает шум | `TASK-058` |
| 3 | Tiebreak: при равной точности — больше верных, затем больше разрешённых | Алфавит/дата | Поощряет активность при равной точности | `TASK-058` |
| 4 | Скоуп — только админ-экран `/leaderboard` | Сразу + бот-команда | Roadmap-фича про интерфейсы админки; бот `/leaderboard` — отдельная будущая задача, чтобы удержать объём S/M | `TASK-058`, [[project_betting_bot]] |
| 5 | Период — MVP all-time; фильтр (30d/90d/all) — опционально | Только all-time; обязательный период | All-time проще и достаточно для старта; период — приятный, но не блокер | `TASK-058` |
| 6 | Агрегат — один SQL `GROUP BY user_id` + `HAVING resolved >= N` + JOIN User | N+1 через `user_stats` по каждому | Один запрос — производительно и совпадает с существующим паттерном `user_stats` | `TASK-058` |

Крупного ADR не требуется — фича переиспользует существующие модель, репозиторий-паттерн и
дизайн-систему; архитектурных развилок нет.
