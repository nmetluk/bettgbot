# Журнал сессий проектирования

В этой папке — журнал работы **cowork-агента** (проектировщика). Каждая сессия — отдельная папка. Цель — чтобы любой, кто откроет проект через месяц, мог восстановить ход мысли: зачем приняли то или иное решение, что было обсуждено, какие альтернативы отвергли.

## Когда заводить сессию

Новая папка сессии — на **каждое подключение проектировщика к проекту**, в котором что-то поменялось в `docs/`, `state/` или `handoff/inbox/`. Если сессия была чисто диалоговой (например, владелец задал вопрос, агент ответил, ничего не изменилось) — записывать не обязательно.

## Naming

```
sessions/YYYY-MM-DD-NN-<kebab-slug>/
```

- `YYYY-MM-DD` — дата начала сессии.
- `NN` — двузначный порядковый номер сессии в этот день (обычно `01`).
- `slug` — короткая тема, до 5 слов: `project-bootstrap`, `data-model-revision`, `bot-flows-draft`.

Пример: `sessions/2026-05-22-01-project-bootstrap/`.

## Структура сессии

```
sessions/YYYY-MM-DD-NN-slug/
├── brief.md              # Зачем была сессия, что просил владелец, что планировалось
├── decisions.md          # Что приняли, что отвергли, с короткой аргументацией
├── transcript.md         # (опционально) Ключевые цитаты из диалога — чтобы восстановить ход
└── artifacts/            # (опционально) Снимки промежуточных артефактов, картинки, эскизы
```

`brief.md` и `decisions.md` — обязательны. Остальное — по необходимости.

## Шаблон brief.md

```markdown
# Brief — <slug>

**Дата:** YYYY-MM-DD
**Длительность:** ~Nч (опционально)
**Участники:** Николай (owner), cowork-agent

## Запрос владельца

Кратко — что попросили, дословно если важно.

## Что планировалось сделать

- …

## Что фактически сделано

- …
- Ссылки на созданные/изменённые файлы.

## Что не сделано / отложено

- … (со ссылкой на BACKLOG.md, если занесено)

## Следующие шаги

- …
```

## Шаблон decisions.md

```markdown
# Решения — <slug>

| # | Решение | Альтернативы | Обоснование | Где зафиксировано |
|---|---|---|---|---|
| 1 | Использовать PostgreSQL | SQLite, MySQL | Нужны индексы по JSONB и транзакции | `docs/adr/0001-tech-stack.md`, `state/DECISIONS.md` |
| 2 | … | | | |
```

Если решение крупное — заводится отдельный ADR в [`docs/adr/`](../docs/adr/). В `decisions.md` сессии всегда даётся ссылка на ADR.

## Принципы

- **Пиши для незнакомого читателя.** Через месяц вы (или другой агент) не вспомните контекст — он должен быть в файле.
- **Решения важнее процесса.** Не нужно протоколировать каждый шаг — только то, что повлияло на проект.
- **Связывай.** Каждый артефакт сессии ссылается на конкретные файлы в `docs/`, `state/`, `handoff/`.
