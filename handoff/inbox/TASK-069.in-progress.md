# TASK-069: BLOCKER — фикс ротации 068 ломает вход для браузеров со «старой» CSRF-кукой

## Контекст (диагностировано вживую на проде после деплоя 068)

TASK-068 задеплоен (подсказка в `login.html` видна; токен формы стабилен между GET'ами — ротация ушла).
**Но логин по-прежнему 403 «Сессия истекла»** — теперь по другой причине, в самом фиксе 068.

Замеры в браузере на `a.pinbetting.ru` (образ с 068):
- Токен формы стабилен между двумя `GET /login` (`tokens_equal: true`) — переиспользование работает.
- Но `POST /login` тем же токеном → **403** даже **немедленно** (0 сек), не только после паузы.
- В свежем (приватном) окне без куки — вход проходит; в окне с накопленной старой кукой — стабильный 403.

**Корень — в `CsrfTokenMiddleware._get_token_from_cookie` (`src/admin/auth/middleware.py`):**
декодирование куки идёт с **`max_age=None`** (строка с `serializer.loads(signed_token, max_age=None)`),
т.е. принимается **просроченная** кука, и в reuse-ветке (`set_new_cookie=False`) **новая кука не
выставляется никогда**. А `validate_csrf` на POST использует `max_age=900` (TTL из `_csrf_config`, TASK-068).

Последствие: как только CSRF-куке в браузере становится **больше 900 c (15 мин)**:
- reuse-ветка всё равно её декодирует (`max_age=None`) и переиспользует токен из неё;
- свежая кука НЕ выдаётся (нет ротации, нет обновления);
- `validate_csrf` (`max_age=900`) считает куку просроченной → `CsrfProtectError` → 403;
- **перезагрузка не лечит** (middleware снова переиспользует ту же просроченную куку).
Браузер залипает в 403 навсегда, пока пользователь вручную не очистит куки. Для возвращающихся
пользователей (а это норма) вход полностью сломан.

## Цель

Вход работает для ЛЮБОГО браузера, в т.ч. со старой/просроченной CSRF-кукой, без ручной очистки кук;
при этом ротация на каждом GET по-прежнему НЕ происходит (регресс 068 не возвращаем). CSRF-защита не ослаблена.
